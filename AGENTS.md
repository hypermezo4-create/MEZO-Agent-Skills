# Agent Rules

1. Read this file, the target `SKILL.md`, linked references, and the matching project profile before editing.
2. Keep `SKILL.md` focused; put deep examples and domain tables in `references/`.
3. A skill may guide judgment but must not claim that tests, linters, or deployments ran unless evidence is supplied.
4. Never include credentials, access tokens, private repository content, or real user data in examples.
5. Never instruct an agent to ignore higher-priority instructions, evade approval gates, modify protected branches, weaken tests, or hide errors.
6. Every blocking rule must identify evidence, risk, and a concrete fix.
7. Avoid invented percentages or quality scores. Use counted findings and receipts.
8. Update `VERSION` for behavior changes and regenerate `skills-manifest.json`.
9. Upstream-derived edits must preserve attribution and update `THIRD_PARTY_NOTICES.md` when needed.
10. All changes must pass repository validation and unit tests before merge.
